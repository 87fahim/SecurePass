Best architecture (recommended)

Keep board editing and move display in your React app.
Add one backend endpoint that takes current FEN and returns the best move.
Use Stockfish as the engine (self-hosted) for strongest and most controllable results.
What to use

Engine: Stockfish 16.1 (or latest).
Backend:
Node + Express + child_process (easy with your current stack).
Or Python + FastAPI + python-chess if you prefer Python.
Position logic on frontend/backend: chess.js for FEN handling and legal move apply.
Third-party options vs self-hosted

Self-hosted Stockfish:
Pros: strongest, configurable depth/time, no API limits, predictable.
Cons: you manage server resources.
Lichess Cloud Eval API:
Pros: quick to integrate.
Cons: rate-limited, less control, not ideal as core product dependency.
Browser-only Stockfish WASM:
Pros: no backend needed.
Cons: heavy CPU on client, weaker devices struggle, less secure for product control.
For your “deepest possible” goal, self-hosted Stockfish is the right choice.

Required product behavior for Free Style setup

User arranges pieces in Free Style.
Validate position before Next Move:
Exactly 1 white king and 1 black king.
Max 8 pawns per side.
Max 16 total pieces per side.
Position must be loadable as FEN.
Enable Next Move button only when valid.
Call backend with FEN and engine settings.
Show suggested move as link text like e2-e4.
On click, apply move in UI with your existing board animation flow.
Engine settings you should expose

movetime (ms): fixed thinking time per request, for example 1000 to 5000.
depth: hard search depth cap, for example 18 to 28.
multipv: usually 1 (best move only), optionally 3 for alternatives.
threads: usually 1 to number of server cores.
hash (MB): for example 128 to 1024 depending on server RAM.
skill level: optional if you want weaker suggestions for beginners.
Suggested API contract

POST /api/chess/next-move
Request:
fen
depth or movetime
multipv (optional)
Response:
bestMoveUci: e2e4
bestMoveText: e2-e4
depth
scoreCp or mate
pv (optional)

Practical recommendation

Start with movetime-based search (more stable UX than very large depth).
Default to movetime 1500 ms, allow “Strong” mode at 5000 ms.
Queue requests and set timeout on backend to avoid server overload.
